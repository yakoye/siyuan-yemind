# YeMind v0.5.11 启动恢复与可编辑大纲设计

## 背景

v0.5.10 的真实诊断包显示：插件启动后 Dock 显示“暂无导图”，恢复标签显示导图不存在；新建任意导图后，旧导图才重新出现。与此同时，分屏和纯大纲模式只提供只读列表，不能像原版界面一样直接修改节点结构与文字。

## 诊断结论

### 启动恢复

诊断事件只有 `plugin/onload`，没有 `plugin/bootstrap-started` 与 `plugin/bootstrap-completed`。自检发生时 `mapCount` 为 0；随后新建导图触发 `MapRepository.ensureLoaded()`，正式存储中的旧导图才被加载并与新导图一起显示。

当前 `onload()` 在启动 `bootstrap()` 之前依次注册 Tab、Dock、Settings、Commands 和事件总线。任何一个同步注册异常都会中止 `onload()`，导致：

1. `ready` 仍是默认已完成 Promise；
2. Dock 和恢复标签误以为仓库已准备好；
3. 正式仓库保持未加载空状态；
4. 新建导图时才被动加载正式数据。

### 大纲编辑

现有大纲由静态 `<button>` 列表渲染，只绑定点击定位，没有编辑控件或键盘状态机。上游 `simple-mind-map` 已提供：

- `SET_NODE_TEXT`
- `INSERT_NODE`
- `INSERT_CHILD_NODE`
- `REMOVE_NODE`
- `GO_TARGET_NODE`

删除命令自身会选择兄弟节点或父节点，因此结构变更必须继续交给上游命令执行。

## 设计目标

### 启动链路

- 核心仓库加载必须先于任何非关键界面注册启动。
- 单个界面注册失败不能阻止仓库、设置和检查点加载。
- `whenReady()` 必须始终代表真实 bootstrap Promise，不能短暂暴露默认已完成 Promise。
- 每个注册步骤写入诊断事件，便于下一次直接定位失败模块。
- 数据 bootstrap 失败仍需明确记录，不能将空仓库伪装为已正常加载。

### 可编辑大纲

- 分屏与纯大纲共用同一套大纲编辑器。
- 单击一行：选择并定位对应导图节点。
- 单击文字：直接进入文字编辑。
- 输入文字实时更新本地输入；失焦、Enter 或方向操作时提交节点文字。
- Enter：在当前节点后创建同级节点并进入新节点编辑；根节点按 Enter 创建子节点。
- Tab：创建子节点并进入编辑。
- Backspace/Delete：仅当当前文字为空时删除当前非根节点；删除后聚焦上一可见行，若不存在则使用下一可见行。
- ArrowUp/ArrowDown：在可见大纲行之间移动。
- Escape：取消本次未提交文字编辑并恢复原文。
- 只读模式下保留选择和定位，但禁用文字与结构编辑。

## 不采用的方案

- 不直接修改 `MindMapTree.children`，避免建立第二套撤销、保存和选择算法。
- 不在大纲中调用隐藏画布 `resize()`。
- 不用固定延时解决启动仓库问题。
- 不复制或反编译原版 KMind Zen 私有内核；公开仓库未提供可稳定复用的编辑器源码。

## 兼容与风险

- 大纲编辑调用上游原生命令，自动进入现有 `data_change`、撤销栈、自动保存、检查点和诊断链路。
- 富文本节点从大纲修改时按纯文本写入，并关闭该节点的 `richText` 标记，避免把未转义文本当 HTML；导图画布内仍可继续使用富文本编辑。
- 结构命令后通过预先指定 UID 聚焦新节点，不依赖渲染时序猜测。
- 启动注册错误会被隔离并记录，但关键 Tab/Dock 注册失败仍会影响对应界面；诊断包将明确指出具体步骤。
